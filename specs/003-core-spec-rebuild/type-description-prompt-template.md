# Type Description Prompt Template

更新时间：2026-06-29

本文档定义 `003-core-spec-rebuild` 的类型描述离线生成模板。

目标不是让模型在运行时自由发挥，而是：

- 用统一模板批量生成 `24+1` 类型文案初稿
- 强制文案体现 `15` 维原型信息
- 强制文案只引用已批准的专业知识卡
- 让每次生成都可审计、可复盘、可迭代

## 1. 适用范围

本模板只用于离线生成每个类型的固定文案资产，例如：

- `intro`
- `sections[]`
- `evidenceTrace`

本模板不用于：

- 运行时用户专属生成
- 医学建议生成
- 心理建议生成
- 去留判断
- 法律或伦理判断

## 2. 设计原则

### 2.1 结构先于文风

先把类型、原型、维度、矛盾和知识卡结构化，再要求文风。

### 2.2 知识卡先于自由回忆

模型只能使用输入里的批准知识卡，不能自由回忆“心理学常识”“社会学事实”或“医学观点”。

### 2.3 模式先于本质

文案必须描述“你更容易进入这种模式”，不能写成“你天生就是这种人”。

### 2.4 审校先于入库

模型输出只是初稿。进入仓库前必须人工审校。

## 3. 生成工作流

推荐流程：

1. 从 `15` 维原型库提取该类型的主高值、次高值、低值和关键张力。
2. 从本地批准的知识卡库中挑选 `2-4` 张相关知识卡。
3. 用本模板生成结构化初稿。
4. 人工检查维度贴合度、专业边界和文风稳定性。
5. 审校通过后再写回类型配置。

## 5. 字段要求

### 5.1 `prototypeVector`

- 必须包含全部 `15` 维
- 只能来自本地原型配置
- 不允许模型自行补值

### 5.2 `dimensionSignals`

这是最关键的中间层。推荐在调用模型前由本地脚本算好。

至少要给出：

- `dominantHigh`
- `notableHigh`
- `notableLow`
- `tensions`

原因：

- 直接喂 `15` 维数值，模型会看但未必抓重点
- 把重点维度和张力显式提炼出来，能降低“只按类型名写”的风险

### 5.3 `dimensionGlossary`

这是维度的短解释层，用来防止模型按英文变量名或日常词义硬猜。

至少应覆盖：

- 全部 `dominantHigh`
- 全部 `notableHigh`
- 全部 `notableLow`
- `tensions` 中涉及的维度

### 5.4 `dimensionNotes`

这是维度的高低说明层，用来告诉模型：

- 该维度高时像什么
- 中间状态像什么
- 低时又像什么

推荐做法：

- 至少为所有重点维度提供 `low / medium / high`
- 这些说明应直接来自本地维度定义，不让模型自行补完

原因：

- `dimensionGlossary` 只能告诉模型“这个维度大概是什么”
- `dimensionNotes` 才能告诉模型“高分和低分分别会怎样落在人的反应里”
- 两层一起给，才能真正降低误解风险

### 5.5 `knowledgeCards`

每次生成推荐 `2-4` 张，过多会稀释重点。

每张知识卡必须只有：

- `id`
- `sourceType`
- `allowedUse`
- `summary`

不直接塞论文原文，不塞大段引用。

### 5.6 `businessContext` 与 `sectionDefinitions`

这两个字段用于强制模型不要只写抽象心理词，而要落到本产品的真实业务语境。

- `businessContext` 用来定义固定场景边界：怀孕、是否继续妊娠、未来养育、与老公或伴侣互动、已有孩子、家庭分工与边界。
- `sectionDefinitions` 用来定义每个标题到底回答什么问题，避免 section 之间内容互相串味。
- 模型必须优先把重点维度翻译成这些家庭场景里的反应，而不是只写“你在意安全感”“你很敏感”“你怕失控”这种抽象句。

## 6. 推荐 System Prompt 模板

```text
你是一个“类型描述文案生成器”，服务于一个围绕怀孕与未来养育想象的娱乐型类型测试产品。

你的任务不是判断用户是否应该继续妊娠，也不是做医学、心理、法律或伦理建议。
你只能基于给定的：
1. 类型基础信息
2. 15维原型向量
3. 维度短解释
4. 维度高低说明
5. 重点维度与张力
6. 已批准的专业知识卡
来生成“该类型的固定说明文案”。

你的写作目标：
- 写出“模式”，不是“本质”
- 让用户感觉“像我”
- 保持幽默、轻微自嘲、娱乐性
- 同时保留专业边界，不病理化，不装专家

你必须遵守以下规则：
- 不得输出任何医学建议、药物建议、手术建议、风险等级、法律结论
- 不得输出心理诊断、人格定性、依恋类型诊断、创伤判断
- 不得暗示“你不适合做母亲”
- 不得发明未提供的维度、事实、研究结论或评分逻辑
- 必须优先使用 `dimensionGlossary` 和 `dimensionNotes` 理解维度，不得按英文变量名自行脑补
- 必须体现输入中最主要的高值/低值/矛盾维度
- 必须体现输入中的知识卡，但只能做轻量转译，不能扩展成专业宣告
- 语言要口语化，有画面感，有一点自嘲，但不过度刻薄

请输出严格符合 schema 的 JSON。
```

## 7. 推荐 User Prompt 模板

```text
请根据下面的结构化输入，为该类型生成固定文案初稿。

写作要求：
1. `intro` 用 1-2 句抓住该类型的主模式，长度控制在 `200-300` 个中文字符
2. `sections` 必须按 `sectionDefinitions` 逐条生成，每段控制在 `100-200` 个中文字符
3. 每个 section 都必须落在本产品业务场景里：发现或确认怀孕、是否继续妊娠、未来养育宝宝、和老公或伴侣沟通、已有孩子与家庭分工
4. `effective_support` 必须写“别人怎么和她互动更有效”，优先写老公、伴侣或家人的沟通与配合方式，不写行动清单
5. 文案不能彼此重复，不能只是把类型名换个说法
6. 必须把重点维度转译成可感知的生活反应，而不是只复述抽象词
7. `evidenceTrace` 只做内部审校用途，不面向用户

结构化输入如下：

{{INPUT_JSON}}
```

## 9. Intro 专项风格约束

当生成 `intro` 时，额外执行以下要求：

- 控制在 `200-300` 个中文字符
- 必须是一整段，不拆成列表
- 开头不能雷同，不能批量以同一句式起手
- 必须保留幽默感、娱乐性和一点轻微毒舌
- 必须像“类型主页开场白”，不能像量表解释或咨询报告
- 必须把重点维度写成生活反应，而不是抽象术语堆叠
- 允许夸张，但不允许病理化或羞辱化

推荐参考口径：

- 不是“你是一个怎样的人”
- 而是“事情一来，你的大脑和反应系统会先怎么动”
- 不是“概念翻译”
- 而是“让用户一眼觉得这段在写自己”

## 10. 推荐审校标准

每次生成后，至少检查以下项目：

1. 是否能从文案里看出 `2-4` 个核心维度，而不是只看见类型名气质。
2. 是否把“模式”写成了“本质”。
3. 是否混入了医学、心理、法律或道德判断。
4. 是否真的使用了知识卡，而不是套壳鸡汤。
5. `intro` 与各个 `sections` 是否互相重复。
6. `effective_support` 是否在写关系语言，而且真的落在老公、伴侣、家人或已有孩子相关的家庭场景里。
7. 是否符合“有梗、有刺、有自嘲，但不恶毒”的边界。

## 11. 推荐知识卡分类

知识卡建议按以下分类维护：

- `psychology`
  用于“混合反应”“防御姿态”“身份张力”的轻量解释
- `sociology`
  用于“照护负担”“关系兑现”“支持方式”“现实分工”的轻量解释
- `medical_boundary`
  只用于解释“身体变化会影响反应”，不能外推成医疗结论

## 12. 推荐项与理由

## 12A. Section 版 Prompt 升级（推荐）

下面这组模板用于当前推荐版本：`intro + 6 个 sections`。  
它的重点不是再多写一点，而是强制模型把文案落进我们的业务场景，而不是写成悬空的人格散文。

### 12A.1 推荐的场景锚点

模型在生成 `intro` 和 `sections` 时，必须优先落在以下场景之一或几种组合里：

- 发现或确认怀孕当下
- 想到是否继续妊娠时
- 想到未来养育宝宝、恢复期、夜里照护、家庭节奏被重排时
- 和老公或伴侣沟通、协商、拉扯、失望或分工时
- 家里已有孩子时，想到精力分配、陪伴公平、手足关系和家庭秩序时
- 长辈介入、家庭边界、家务与照护承重被重新分配时

禁止只写抽象句式，例如：

- “你很缺安全感”
- “你很在意边界”
- “你很害怕失控”

必须把这些抽象点翻译成家庭场景里的具体反应。

### 12A.2 推荐的 Section 写作目标

针对不同 section，请遵守以下写作目标：

- `first_reaction`：写她在发现或确认怀孕当下，注意力先扑向哪里，通常是先想身体、先想老公怎么反应、先想已有孩子，还是先想现实安排。
- `mental_preview`：写她想到是否继续妊娠、未来带宝宝、恢复期、夫妻分工、家庭节奏和已有孩子时，脑内最常先预演哪类剧情、风险、关系走向或生活后果。
- `core_concern`：写她表面像在纠结 A，其实真正卡住的是哪类心理、关系、家庭分工、身份变化或现实承重点。
- `self_pressure`：写她在怀孕、未来养育想象、和老公协商、顾及已有孩子或维持家里运转时，最容易怎样把自己顶上去、扛过头或把压力内化。
- `distortion_trigger`：写她在什么家庭场景下最容易明显变形、失真、过载、退避、控场、炸毛或装没事。
- `effective_support`：写老公、伴侣或家人怎样和她说话、怎样分担、怎样给支持，她会更容易放下防御，而不是更想自己一个人顶着。

### 12A.3 推荐的 Section Prompt 追加约束

```text

总原则：
1. 每个 section body 控制在 100-200 个中文字符。
2. 必须是一整段，不分点。
3. 必须同时参考：
   - prototypeVector
   - dimensionSignals
   - dimensionGlossary
   - dimensionNotes
   - knowledgeCards
   - businessContext
   - sectionDefinitions
4. 每个 section 都必须落在本产品业务场景里，优先写：
   - 发现或确认怀孕当下
   - 是否继续妊娠时的拉扯
   - 未来养育宝宝与恢复期
   - 和老公或伴侣沟通、分工、失望或互相配合
   - 家里已有孩子时的精力、秩序和公平
5. 必须把维度翻译成生活化反应，不能直接复述变量名。
6. 允许幽默、自嘲、轻微毒舌、娱乐感，但不能病理化、羞耻化、说教。

8. 不能与 intro 重复，section 之间也不能重复。

- 要像“懂她的人写给她看的爆款解释”，不是量表说明书。
- 每段都要有一个可感知的小画面或小机制。
- 每段最好带一点黑色幽默或轻微自嘲感，但不要刻薄。
```

### 12.1 推荐保留 `evidenceTrace`

理由：

- 方便审校时判断模型有没有真的用到正确维度
- 便于后续比较不同 prompt 版本的失误模式

### 12.2 推荐把 `dimensionGlossary` 和 `dimensionNotes` 一起喂给模型

理由：

- 只给维度名，模型很容易按字面硬猜
- 只给短标签，不足以说明高低变化的含义
- 两层同时存在时，模型更容易把维度写成生活化反应，而不是抽象人格词

### 12.3 推荐由本地脚本先计算 `dimensionSignals`

理由：

- 模型擅长写，不擅长稳定抓重点
- 重点维度先抽出来，能明显降低空泛输出

### 12.4 推荐离线批量生成再人工定稿

理由：

- 最符合当前项目“本地规则 + 模板”的方向
- 不引入运行时不稳定性
- 容易做内容版本管理

## 13. 不推荐项

- 不推荐直接让模型运行时按用户答案自由生成
- 不推荐只给一句“请结合15维和专业知识写文案”的大 prompt
- 不推荐把论文原文大段塞进 prompt
- 不推荐只给维度英文名，不给解释层和高低说明层

这些做法都会提高空泛输出、越界输出和不可审计输出的概率。
