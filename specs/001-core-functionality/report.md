# Report Spec

更新时间：2026-06-26

本文档定义 `should-we-continue` 的报告信息架构、显示优先级、共享边界与降级行为。它不定义红旗阈值、评分权重或角色映射；这些分别由 `specs/scoring.md` 和 `specs/personas.md` 定义。

依赖文档：

- `specs/research.md`
- `specs/questionnaire.md`
- `specs/scoring.md`
- `specs/personas.md`

## 1. 报告目标与原则

报告是决策支持报告，不是“继续妊娠 / 终止妊娠”的推荐器。它应帮助用户按以下顺序看清局面：

1. 是否有需要优先线下处理的医疗、心理、安全或法律问题。
2. 此刻的决策风格、支持需求和不确定性来自哪里。
3. 九个支持维度分别处于什么等级，哪些信息仍缺失。
4. 两条路径各自需要确认或补齐什么条件。
5. 哪些事情能由自己完成，哪些需要伴侣、家庭或专业人士共同承担。

用户不可见总分、路径分数、人格百分位、适配度、排名或“系统建议你选择哪一条路”。

最终个人报告默认应在 2-3 分钟内读完核心内容；详细原因、条件解释、来源卡片和延伸建议使用可展开区域承载。报告不回显用户的原始选择题回答，也不提供单独的“修改答案”入口。

视觉基线为浅色模式与系统中文字体栈；不得因未加载外部字体而改变排版、信息层级或阅读功能。

## 2. 页面与访问关系

报告固定为三个页面：

1. **概览**：个人报告的首页，先帮助用户理解“此刻的自己”，再展示九维全貌。
2. **深入分析**：先展示两条路径共同需要确认的事实，再并列呈现继续妊娠和终止妊娠各自的条件清单。
3. **伴侣讨论**：仅在用户主动选择共享内容后可用，产出三个共同讨论议题和可执行承诺清单。

前两个页面始终属于用户个人报告。第三个页面不是个人报告数据的默认镜像，而是一次基于授权内容重新生成的讨论材料。

移动端和桌面端均使用页面顶部的固定标签切换：`概览 / 深入分析 / 伴侣讨论`。伴侣讨论标签在用户未生成共同讨论页时显示为不可用状态，不自动跳转或隐藏其他个人报告页面。

无 R3/R4 时的默认顺序为：核心问卷完成 → 显示 2-3 个深入模块推荐及额外的 12 题角色化类型细化 → 用户完成推荐模块或明确跳过 → 生成三页最终个人报告。MVP 不在核心问卷完成后先生成常规基础报告。

## 3. 全局优先级与安全替代路径

### 3.1 R3/R4

R3/R4 时，不渲染常规三页结构作为首屏。替代为“线下支持优先”页面，按最高红旗等级展示：

- 清晰、克制地说明当前需要优先获得线下医疗、心理、法律或安全支持。
- 显示最少、最直接的下一步行动，不展示角色、状态标签、趣味插画、路径比较或伴侣承诺清单。
- MVP 红旗提示只使用简短的通用线下处理说明，不展示电话号码、地区机构卡片、电话簿或本地资源列表。
- 保留返回、清除本机数据和在安全条件下继续填写的能力，但不鼓励先完成报告再求助。
- 涉及胁迫、暴力、无法安全填写或分享风险时，禁用伴侣讨论页入口；不能因用户先前的分享勾选而自动生成共享材料。

### 3.2 R1/R2

R1/R2 不阻断常规报告，但相关提醒必须在涉及模块之前显示，并使用中性、非趣味化语言。角色模块如显示，须位于风险提示和优先行动之后。

### 3.3 信息不足与低确定性

- 任一维度 `certaintyLevel = low` 时，维度等级后显示“仍需确认”，并链接到推荐的深入模块或行动项。
- 12 题角色化类型细化未达到最少答题量、候选差异过小或映射置信度不足时，角色区域显示“仍在校准中”，不强行展示主角色。
- 缺失不能被解释为低支持，也不能由大模型补全。

## 4. 页面一：概览

无 R3/R4 时，概览页按照以下顺序显示。

### 4.1 角色与状态标签首屏

首屏优先显示：

- 主角色固定插画或降级图标、称号和一句概括。
- 次角色（如规则服务返回）使用小图标、称号和一句补充的支持需求；不与主角色并列展示完整插画。
- 5-7 个最有指向性的状态标签。
- 一句中性提示：角色描述的是当前的决策支持需求，不是生育适配性或人格诊断。

角色插画、图标、文本和 R3/R4 抑制规则必须遵循 `specs/personas.md`。

当前 MVP 的正式量表均为 `unavailable`，因此角色区域之后不渲染“测量明细”、量表名称、维度、原始分数或互动风格标签。未完成 12 题角色化类型细化时，只显示继续完成入口和“仍在校准中”的中性提示。

### 4.2 九维支持全貌

角色区域之后，展示九个支持维度的等级总览。桌面端可用固定 3 × 3 网格；移动端使用同顺序的单列紧凑列表。每项只显示：维度名称、用户可见等级、确定性提示和一行“为什么需要关注/目前较稳定”的摘要；详细原因收纳在可展开区域。

九个维度：

1. 医学与身体安全
2. 心理与情绪承载
3. 自主与安全空间
4. 个人意愿清晰度
5. 人生发展与节奏
6. 伴侣承诺与共同承担
7. 家庭与社会支持
8. 经济与政策准备
9. 已有子女与照护负荷

`lifeDevelopmentSupport` 必须有独立高亮，不能被其他维度的良好状态淡化。所有等级映射使用 `scoring.md` 的固定文案，不展示 `supportScore` 数值。

九维支持全貌不使用雷达图、柱状图、热度图或其他图表，只使用文字等级、稳定图标和简短原因，避免制造精确量化的错觉。

心理筛查相关结果只可显示为非诊断性的支持提示，例如“情绪与支持需要优先关注”或“情绪信息尚不完整”；不得显示 `PHQ-2`、`GAD-2`、`PHQ-9` 或 `GAD-7` 的原始分数、诊断名称或疾病标签。

### 4.3 优先行动与深入建议

九维总览之后显示：

- 当前最需要确认的三件事，遵循 `scoring.md` 的报告排序规则。
- 2-3 个推荐深入模块及其原因。
- 用户若先前跳过深入模块，可从首页的继续填写流程补充后重新生成最终报告；报告页面本身不提供单独修改答案入口。

### 4.4 伴侣视角（可选）

当伴侣主动分享互动风格摘要后，重新生成用户个人报告，并在优先行动之后展示“伴侣视角”区。该区只呈现已授权的沟通偏好、支持方式和愿意讨论事项，并清楚标注其来自伴侣自述。

伴侣视角只帮助用户理解共同讨论可能需要怎样展开；它不得改变个人报告的红旗、九维等级、确定性、主/次角色、状态标签或两条路径条件清单，也不覆盖用户本人的意愿。

## 5. 页面二：深入分析

深入分析的目标是减少短视决策，不把用户推向任一条路径。内容分为“共同事实”与“并列条件”两部分。

### 5.1 共同事实先行

任何路径比较之前，先显示两条路径都需要面对的内容：

- 医学信息是否完整，以及哪些问题只能由线下专业人员确认。
- 是否存在自主、安全、时间或信息完整度问题。
- 本人的真实意愿是否已能和外界期待区分。
- 当前最主要的价值冲突、信息缺口和长期担忧。
- 以 1 周、1 个月、1 年、5 年为时间点的复盘提示，帮助用户看见短期和长期支持需求。

该区域不解释检查结果、不提供药物或手术建议，也不把医学信息包装成路径推荐。

在经济、政策和照护相关的深入分析区域，可显示杭州滨江区地区配置提供的简短政策、资源和成本参考卡片。每张卡片必须显示来源链接、最后核对日期、适用条件摘要和“请按本人单位/经办口径确认”的限制说明；来源链接在新标签页打开。成本参考只用于帮助用户补充预算，允许修改或忽略，不得替代个人实际预算或决定路径条件。MVP 不提供地区切换；超出该地区适用范围时仅保留通用清单与用户预算输入。

用户可主动点击“检查最新地区信息”，触发运行时模型辅助检索。检索只可更新带完整来源证据并通过 schema 校验的本地卡片；检索失败、来源非官方白名单、缺少最后核对日期或字段校验失败时，报告不展示候选内容，而是退回通用清单。

已验证的本地卡片在当前浏览器缓存 7 天。缓存超过 7 天而用户尚未重新检查时，卡片可继续显示，但必须标注“可能过期”并保留来源链接和最后核对日期。官方白名单来源相互冲突时，并列展示冲突内容及各自来源和日期，不自动裁决，不用于评分或路径条件。

### 5.2 两条路径的并列条件清单

共同事实之后，展示两个等权重的条件区域：

- **如果继续妊娠，需要确认或补齐的条件**。
- **如果终止妊娠，需要确认或补齐的条件**。

每个区域使用相同的结构：

1. 必须优先确认的事项。
2. 当前已有的支持条件。
3. 仍需补齐的支持、照护、沟通或专业咨询条件。
4. 决定后的身体、心理、关系和生活安排提示。

两侧不显示分数、成功率、倾向箭头、默认选中状态或“更适合”的视觉暗示。两条路径的条件默认全部折叠，用户主动打开后阅读；每侧标题显示不带倾向含义的进度计数，例如“已确认 2/6”；桌面端并列呈现，移动端采用同层级的连续区块或切换标签，且标题、项数和视觉权重保持一致。

每个条件条目允许用户标记当前进度：`已确认`、`待确认`、`暂不处理`。该状态只用于组织后续行动和报告阅读，不自动加减 `supportScore`、改变红旗或形成路径推荐；用户需要修改事实性回答时，必须返回对应问卷题目并重新计算。

### 5.3 深入分析总体备注

深入分析页的两条路径条件清单之后，提供一段简短的总体备注，用于补充跨条件的现实安排、尚未确认的事实或希望带去线下咨询的问题。MVP 不为每条条件提供独立备注。

备注默认属于个人报告，不自动进入伴侣讨论页或大模型输入。如需用于共同讨论，用户必须先编辑一份独立的分享摘要；原备注始终保留在个人报告中，不得原样复制到伴侣页。分享摘要只在用户明确授权后，作为共同讨论议题的补充上下文使用。

## 6. 页面三：伴侣讨论

### 6.1 进入前的授权步骤

进入该页面前，用户必须逐项选择可分享内容。默认不分享：

- 自伤、自杀、暴力、胁迫和安全风险相关回答。
- 个人真实倾向、自由文本、医疗细节、完整量表原始答案和人格维度。
- 未明确勾选的任何答案或报告摘要。

具体分享范围只在个人报告生成后选择，可多选：医学信息、情绪压力、个人发展顾虑、经济压力、伴侣承诺需求、家庭支持和边界、已有子女照护、价值观和遗憾，以及是否展示两条路径的条件清单。即使选择分享条件清单，未授权的个人原始回答、医疗细节、真实倾向、自由文本和安全风险内容仍不得自动共享。

用户可以只分享经过模板改写的议题，不分享原始回答。对于深入分析总体备注，必须由用户编辑分享摘要，不能将原备注自动改写或直接带入。每次重新生成共同讨论页前均再次展示分享范围。

伴侣也必须主动选择可分享的互动摘要。用户个人报告已存在时，伴侣共享摘要会先刷新其中的“伴侣视角”区；用户可以在生成或更新共同讨论页时查看双方已授权的摘要范围。共同讨论页可先仅根据用户已授权内容生成，伴侣日后分享后再本地重新生成。

伴侣测评入口不出现在首页。用户完成本页的个人分享选择并生成初始共同讨论页后，才可选择“交给伴侣填写”。该操作进入独立的伴侣测评屏，不先展示用户个人报告的内容；伴侣完成、查看自己的摘要并主动分享后，系统在同一浏览器本地刷新共同讨论页和用户个人报告中的折叠“伴侣视角”区。伴侣测评始终可跳过。

若用户选择在共同讨论页展示两条路径的完整条件清单，必须有单独的“分享两条路径条件清单”授权项。共同页中的“完整”是指完整展示基于**已授权信息**生成的两个条件区域，不等于公开所有个人原始回答。任何未授权内容必须以中性占位提示呈现，例如“此项由本人自行确认”，不得显示原文、数值或敏感原因。

### 6.2 页面产物

伴侣讨论页产出以下三类材料：

1. **三个共同讨论议题**：从已授权内容、低支持维度和未确认承诺中选择，使用“需要一起确认什么”而非“谁做得不够好”的语言。
2. **简短承诺清单**：围绕医学陪伴、经济承担、时间与劳动、边界保护、情绪支持、长期照护、已有子女照护和决定后支持等预定义类别，列出少量可勾选的具体事项。
3. **两条路径的完整条件清单**：仅在用户勾选对应分享授权后显示，保留个人深入分析页中的对称结构和非推荐措辞。

每个承诺项必须是可讨论、可勾选的具体事项；不得用伴侣人格标签、依恋维度、用户私密回答或系统对双方关系的评价替代承诺内容。MVP 不提供提醒、负责人分配、完成期限、完成核验、双向确认或长期追踪；勾选仅服务于当次讨论，不可被解释为用户已同意某条妊娠路径。

### 6.3 不生成的内容

伴侣讨论页不得：

- 给出双方谁“更适合”继续或终止的结论。
- 默认展示任何一方的角色、正式量表结果或原始文本。
- 把用户的身体自主权转移为共同投票或协商裁决。
- 在存在安全风险时生成可被他人查看的讨论材料。

## 7. 模板、大模型与降级

报告由本地规则先生成结构化数据和模板。用户点击“生成报告”或“重新生成报告”时，如已配置 API Key，系统自动使用大模型完成受控润色；大模型输入和输出必须遵守 `AGENTS.md`、`scoring.md` 与 `personas.md` 的限制。

- 本地规则决定红旗、维度等级、优先事项、路径条件、角色 ID、状态标签和共享范围。
- 便宜模型只处理预设标签和矛盾检查；较强模型只生成最终报告叙述。
- 个人报告请求可包含用户自己的自由文本和深入分析总体备注，作为待解释资料；这些文本不能覆盖系统指令、触发新规则或自动进入伴侣讨论请求。
- 大模型只能润色已选内容、把已授权的摘要转写为讨论议题，或在预设模板内生成自然语言。
- 无 API Key、调用失败、超时或 schema 校验失败时，静默使用规则模板报告；不得丢失红旗、维度、行动项、路径条件或共享边界。首次模型调用失败后最多自动重试 3 次，不向用户显示调用次数、错误或 AI 润色标识。
- 模型输出不得新增医学事实、政策事实、评分、角色名称、风险等级或承诺类别。

## 8. 本地保存与导出

- 用户修改任意问卷回答后，在其主动重新生成报告时覆盖当前浏览器本地保存的报告；MVP 不保留报告版本历史、命名快照或比较视图。
- 用户修改会影响报告的回答后，报告显示“需要更新”状态；只有用户主动点击“重新生成报告”时，才重新运行规则计算和可选的 AI 润色，不在每次回答变动后自动调用模型。若回答未变化，点击该入口直接展示当前报告，不重新调用模型。
- 个人报告支持导出为纯文本或 Markdown，默认包含该个人报告中的自由文本和深入分析总体备注，供用户本地留存或自行使用。
- 导出文件默认只来自个人报告；伴侣讨论页如需导出，必须作为独立内容并再次确认分享范围。
- 导出不包含内部 `supportScore`、权重、红旗规则实现细节、正式量表原始作答或未授权内容。
- 检测到本地已有数据时，首页显示三个明确入口：重新填写并更新报告、查看最新个人报告、清除本机数据；不自动跳转到问卷或报告。重新填写保留已有答案供修改，新的报告在用户主动生成时覆盖旧报告。选择清除时，用户必须在“仅用户数据 / 仅伴侣数据 / 全部本机数据”中选择范围后确认。
- 检测到项目本地数据结构版本不兼容时，显示简短告知后清除旧版项目数据；MVP 不提供迁移、旧数据导出或兼容读取。清除仅针对本项目在当前浏览器保存的数据。
- 自由文本在个人报告中只显示通过受控规则或模型生成的摘要，不显示原文；模型不可用、摘要校验失败或用户未填写文本时，直接隐藏该摘要区域。原文仅保留在本地数据和用户主动导出的个人文件中。
- MVP 不接入使用分析、行为分析或第三方错误监控服务；开发期诊断仅限 `AGENTS.md` 规定的本地/开发环境面板。

## 9. Phase 0 稳定内容注册表

本节是模板渲染和模型叙述覆盖层的唯一内容目录，并覆盖前文未编号的示例文案。模板、模型和 UI 只能请求下列 `contentId`，不能临场发明医学、政策、角色、承诺类别、风险行动或路径结论。全部 `contentId` 的文本均须按本节固定语义呈现；模型可做等义、非诊断性润色，但输出必须仍引用原 `contentId` 与允许变量。

### 9.1 全局允许变量与禁止字段

| 内容类别 | 允许变量 |
| --- | --- |
| 安全行动 | `actionId`, `redFlagLevel`, `supportCategory` |
| 维度摘要 | `dimensionId`, `visibleLevel`, `certaintyLevel`, `reasonId`, `recommendedModuleId` |
| 角色 | `primaryPersonaId`, `secondaryPersonaId`, `personaConfidenceBand`, `statusTagIds`，且仅当 `suppressedReason=null` |
| 路径条件 | `conditionId`, `derivedStatus`, `readingStatus`, `actionCategoryId` |
| 地区卡片 | `regionCardId`, `sourceUrl`, `verifiedAt`, `eligibilitySummary`, `uncertaintySummary`, `freshnessState` |
| 共享讨论 | `shareCategoryId`, `discussionTopicId`, `commitmentCategoryId`, `authorizedSummaryId` |

报告视图模型、模板变量和模型覆盖输出**不得**包含：任何原始回答、`supportScore`、内部权重、路径覆盖度、去留倾向、原始自由文本、原始日期/金额、红旗事实细节、正式量表题项或分数、未授权伴侣数据、提示词、密钥、模型原始输出或重试信息。自由文本只可在个人报告请求的不可执行资料区使用；即使模型成功，也只能返回已通过 schema 的 `summaryId`，不能将原文放进视图模型或共同讨论请求。

### 9.4 共享授权、讨论与占位文本

共享分类固定为：`medical_summary`、`emotional_summary`、`life_summary`、`financial_summary`、`partner_needs`、`family_boundary_summary`、`childcare_summary`、`values_summary`、`path_conditions`、`edited_note_summary`。所有分类默认 `false`，并且 `safety`、`will_direction`、`free_text_raw`、`medical_detail`、`measure_data` 永远不可作为共享分类。

| contentId | 允许变量 | 固定用途 |
| --- | --- | --- |

### 9.5 模型覆盖层与最少测试

模型覆盖层输入只能接收上述 `contentId`、允许变量和其自身的可选个人资料数据块；输出 schema 为 `sectionId`、`contentId`、`variables`、`narrative`。`sectionId` 与 `contentId` 必须来自注册表，`variables` 必须是对应白名单的子集，`narrative` 不得包含新医学/政策事实、角色、承诺类别、风险等级、路径建议、原始文本或数值。任何 schema 失败均丢弃覆盖层并使用同一 `contentId` 的模板。

最少测试场景：R4/R3 仅渲染允许的安全 ID；R1/R2 位于普通段落前；九维无数值、无总分、无路径推荐；`lifeDevelopmentSupport < 50` 使用 `ACT-LIFE-DEVELOPMENT-PRIORITY`；低确定性使用 `RPT-LOW-CERTAINTY`；未授权内容只显示占位；量表明细缺席；模型不可用仍生成完整模板；修改答案使报告过期且导出不含内部或未授权字段。
