# should-we-continue MVP Spec

## 1. 产品定位

`should-we-continue` 是一个面向早早孕女性的网页问卷与报告工具，用于辅助她自我评估是否继续妊娠或中止妊娠。

用户可见产品名称固定为 `Should We Continue`。

产品输出必须定位为“决策支持报告”，而不是“决策结论”。它帮助用户看清医学安全、心理状态、个人意愿、伴侣承担、家庭支持、经济条件、已有子女负担、价值观冲突和下一步行动，但不得替用户决定。

## 2. 当前 MVP 用户与地区

- 主要目标用户：当前先服务一个具体用户，即用户的妻子。
- 默认地区：浙江省杭州市滨江区。
- 地区能力：MVP 固定使用浙江省杭州市滨江区的地区配置，不在问卷或设置中询问、展示或允许修改地区。每个地区配置字段必须附带来源链接和最后核对日期。
- 适用孕周：孕周超过 12 周的用户仍可完成问卷和报告；医学信息区应明确提示尽快线下确认适用的医疗与决策流程。
- 伴侣范围：MVP 假定用户有已识别且可联系的伴侣，不单独设计无伴侣或伴侣未知的个人流程；产品文案统一使用中性的“伴侣”。

## 3. 产品原则

- 决策自主：工具不输出“应该继续”或“应该中止”。
- 安全优先：红旗情况优先于所有评分、角色化和趣味表达。
- 专业约束：评估框架必须依据医学、心理学、社会学、经济学和本地政策调研，不得臆想。
- 隐私优先：第一阶段个人填写内容默认不进入伴侣共同版。
- 轻量实现：MVP 保持简单，不做账号、云端数据库或长期历史记录。
- 大模型受控：模型只在结构化框架内生成表达，不自由发明规则和结论。
- 趣味有边界：风格可参考 SBTI / MBTI 的活泼角色化，但严肃内容必须克制。

## 4. 核心用户流程

### 4.1 第一阶段：个人私密填写

用户单独完成问卷。该阶段重点收集真实感受、医学基础信息、心理压力、个人意愿、伴侣关系、家庭支持、经济准备、已有子女负担和价值观冲突。

该阶段内容默认只用于个人报告，不进入伴侣共同版。

无 R3/R4 时，用户完成核心问卷后先进入深入模块推荐页。系统推荐 2-3 个最值得补充的模块，其中可包含 96 题完整角色测评；用户完成推荐模块或明确跳过后，才生成最终个人报告。MVP 不在核心问卷结束时直接生成常规基础报告。

### 4.2 个人报告

系统生成个人决策支持报告，包含：

- 红旗提示与线下求助建议。
- 当前最需要确认的 3 件事。
- 医学信息完整度和需要找医生确认的问题。
- 当前支持继续妊娠的因素。
- 当前支持暂缓、重新评估或咨询专业人士的因素。
- 如果继续妊娠，需要补齐的支持条件。
- 如果中止妊娠，需要关注的身体、心理和支持条件。
- 个人意愿、价值冲突和资源缺口。
- 角色化理解标签，用于帮助理解沟通风格和不确定性处理方式。

报告不得展示总分或“适合度”分数。

### 4.3 第二阶段：伴侣共同讨论版

用户在个人报告中主动勾选愿意分享的内容。

系统据此生成伴侣共同讨论版，包含：

- 可分享议题。
- 伴侣需要明确承诺和承担的事项。
- 沟通清单。
- 下一步行动清单。
- 未决问题和需要共同确认的信息。
- 仅在用户单独授权后展示的两条路径条件清单；该清单只能基于已授权内容生成。
- 可选的“交给伴侣填写”入口：伴侣完成独立测评并主动分享摘要后，可本地刷新共同讨论页和用户个人报告中的伴侣视角。

伴侣共同版只能包含用户和伴侣各自主动勾选的内容。敏感个人回答不得默认进入共同版。

## 5. 必须评估的一级维度

MVP 的评估框架包含 9 个一级维度：

1. 医学与身体风险：孕周、既往病史、当前症状、医生建议。
2. 心理健康与情绪承载力：焦虑、抑郁、恐惧、期待、压力来源。
3. 个人意愿清晰度：能否区分本人想法、外界期待、信息不足、价值冲突和时间压力；不评价她更倾向哪一条路径。
4. 个人发展与人生规划：本人对生育、事业、自由、家庭和身份变化的真实偏好。
5. 伴侣关系与共同承担：关系稳定性、沟通质量、育儿分工、经济承诺。
6. 家庭与社会支持：双方父母、朋友、照护资源、冲突和边界。
7. 经济与城市政策环境：收入、储蓄、债务、住房、城市、生育政策、托育成本。
8. 已有子女与照护负担：已有孩子数量、年龄、特殊需求、精力分配。
9. 价值观与不可量化因素：宗教或伦理观、遗憾承受方式、生命观、遗憾承受、未来生育期待和不可接受底线。

深入问卷的重点是减少短视决策。它应把决策冲突、信息缺口、决定后支持计划、未来生育与避孕规划融合进相关模块，而不是作为孤立的趣味或附加功能：

- 决策冲突与信息缺口应融合进个人意愿、医学信息、伴侣承诺和报告行动清单。
- 决定后的支持计划应融合进继续妊娠路径和中止妊娠路径的条件清单。
- 未来生育与避孕规划应融合进价值观、人生节奏和医学咨询行动清单。
- 深入问卷不追求题量堆叠，而是帮助用户看到 1 周、1 个月、3 个月、1 年后的现实影响和支持需求。

## 6. 红旗规则

一旦命中红旗情况，报告必须优先提示立即寻求线下医疗、心理、法律或安全支持，并暂缓继续做价值排序或趣味化人格结论。

红旗至少包括：

- 疑似宫外孕。
- 剧烈腹痛。
- 大量或异常出血。
- 自伤或自杀念头。
- 被伴侣、家人或外界胁迫。
- 家庭暴力或人身安全风险。
- 无法自由选择。
- 严重抑郁、焦虑或精神状态失控。

MVP 不收集年龄或生日，因此未成年人妊娠属于当前无法主动筛查的已知覆盖缺口；不得将本产品描述为已覆盖或排除了该风险。后续如纳入年龄段题，才可将其加入自动红旗规则。

具体触发条件、措辞和行动建议必须在 `specs/scoring.md` 中定义，并引用 `specs/research.md` 的依据。

## 7. 医学信息边界

问卷应纳入医学基础信息：

- 孕周或末次月经日期。
- 是否已做早孕试纸、验血或 B 超确认。
- 是否确认宫内妊娠。
- 当前是否有腹痛、出血、发热、晕厥等症状。
- 既往重大疾病、流产史、分娩史或医生已提示的风险。

这些信息只用于：

- 判断信息是否足够可靠。
- 识别红旗。
- 提醒哪些问题必须找医生确认。
- 区分“尚未确认怀孕”“已确认怀孕但未确认宫内”“已确认宫内妊娠”等状态。

工具不得提供具体药物、手术或治疗建议。

## 8. 报告表达

报告采用“个人报告三页 + 伴侣讨论页授权生成”的结构：

- 概览：无红旗时先呈现角色称号和状态标签，再呈现九维等级总览、优先行动和深入模块建议。
- 深入分析：先呈现两条路径共同需要确认的医学、安全、意愿和长期影响事实，再并列呈现两条路径各自的条件清单。
- 伴侣讨论：只呈现用户主动选择分享的内容，并转换为三个共同讨论议题和可执行承诺清单。

用户可见表达采用多维度等级和条件清单，不展示总分：

- 医学信息完整度：低 / 中 / 高。
- 当前安全风险：低 / 中 / 高 / 红旗。
- 心理承载压力：低 / 中 / 高。
- 伴侣支持确定性：低 / 中 / 高。
- 经济准备度：低 / 中 / 高。
- 育儿支持网络：低 / 中 / 高。
- 个人意愿清晰度：低 / 中 / 高。
- 决策时间压力：低 / 中 / 高。

上述限制针对 `supportScore`、路径条件和生育适配性。用户个人报告中可在折叠的正式测量明细区显示已完成量表的名称、维度、原始分数、该量表的原始分数范围、版本和来源链接，但不得把它们合成为总体人格分、诊断或妊娠去留建议。伴侣仅在自己的摘要中查看同等明细；除非其另行授权，不进入共同讨论页。

内部可以使用权重和分数做等级、聚类和角色化映射，但不得向用户显示总分、排名或“适合度”。

## 9. 角色化结论

角色化结论只用于帮助用户理解自己的风格与沟通偏好，不作为妊娠去留的决定依据。

角色化结论可以表达：

- 面对不确定性时更需要数据、情感支持、时间独处还是关系确认。
- 更在意自由、稳定、成就、亲密、家庭延续、身体安全中的哪些因素。
- 与伴侣在沟通风格、风险承受和育儿想象上的差异。
- 一个温和幽默的角色形象和昵称。

角色类型、形象、映射规则和可用文案边界必须提前在 `specs/personas.md` 中定义。大模型只能从预设类型中选择和润色，不得自由创造新的类型体系。

## 10. 明确移除的功能

MVP 不包含六爻、梅花易数、占卜、预测或类似功能。

原因：

- 当前产品需要优先建立医学、心理、关系、经济和决策支持的专业可信度。
- 占卜类内容容易被误解为预测或决定依据。
- 用户对不确定性的处理应通过信息缺口、价值澄清、专业咨询和现实支持计划完成。

## 11. 大模型能力

大模型是报告生成与表达优化的重要增强能力，但必须在受控框架中工作。

允许使用模型的环节：

- 根据结构化结果生成温柔、可读的个人报告。
- 将用户主动勾选可分享的内容改写为伴侣共同讨论版。
- 解释本地规则已选定的角色类型，并在预设文案范围内生成沟通建议。
- 对“不确定”或矛盾回答生成澄清问题。

禁止使用模型的环节：

- 直接判断是否继续妊娠或中止妊娠。
- 自由生成医学、法律、政策或经济事实。
- 自由发明评分规则、问卷维度、红旗规则或角色体系。
- 替代本地红旗规则进行安全判断。

配置策略：

- 使用 OpenAI-compatible 接口，不绑定单一供应商。
- 本地使用 `.env.local`。
- Vercel 部署使用项目环境变量。
- 前端不得直接持有 API Key。
- 无 Key 时降级到规则模板报告。

建议配置项：

- `LLM_PROVIDER_NAME`
- `LLM_API_KEY`
- `LLM_BASE_URL`
- `LLM_MODEL_REPORT`
- `LLM_MODEL_ANALYSIS`
- `LLM_TIMEOUT_MS`
- `LLM_MAX_OUTPUT_TOKENS`

模型策略：

- 用户主动点击“生成报告”或“重新生成报告”时，如已配置 API Key，自动执行受控模型流程；不在每次修改回答后自动调用。
- 单次生成在首次模型调用失败后，最多自动重试 3 次；重试退避、超时和成本护栏由服务端配置控制，不向用户提供调用次数设置。
- 预设标签和矛盾检查：使用便宜模型，由 `LLM_MODEL_ANALYSIS` 配置。
- 最终报告生成：质量优先，使用中等或强模型，由 `LLM_MODEL_REPORT` 配置。
- 角色文案：只能解释本地选定的角色，可复用最终报告模型或规则模板。
- 地区检索：仅在用户手动触发时复用 `LLM_MODEL_REPORT`，以提高对官方字段和适用条件的提取质量。
- 红旗判断：不用模型。
- 医学、政策、经济事实：不用模型自由生成。
- 地区政策与资源：用户主动点击检查最新地区信息时，允许模型从地区配置的官方白名单来源提取候选字段。每项必须带来源链接、最后核对日期和适用条件，并通过固定 schema 校验；缺失、校验失败或检索失败时隐藏本地卡片，退回通用清单。已验证字段缓存于当前浏览器 7 天；超过 7 天但尚未重新检查时，可继续显示但必须标注可能过期。发生官方来源冲突时并列展示各自来源和日期，不由系统自动裁决。

## 12. 隐私策略

MVP 采用本地优先隐私策略：

- 默认不上传、不建账号、不使用云端数据库。
- 默认保存全部回答到当前浏览器本地存储，不上传云端；用户可导出，并可分别清除用户数据、伴侣数据或全部本机数据。
- 本地数据结构版本不兼容时，MVP 在简短告知后清除该浏览器中的旧版项目数据，不做迁移、旧数据导出或兼容读取。
- 不记录敏感日志。
- MVP 不接入使用分析、行为分析或第三方错误监控服务。
- API Key 只存在本地配置或服务端环境变量中。
- 不在产品界面额外要求用户确认自由文本将用于个人报告的模型请求，也不在报告中标识 AI 润色来源；请求范围仍受既有本地、共享和模型输入边界限制。
- 生成伴侣共同版前，必须再次确认哪些内容允许分享。

## 13. 技术方案

默认技术栈：

- TypeScript
- React + Vite
- Tailwind CSS
- lucide-react
- Vitest
- npm
- Vercel 静态部署优先
- 大模型通过 Vercel Functions 服务端代理调用
- 仅开发环境提供 AI 诊断面板，显示配置是否可用、模板降级状态和错误摘要；不得显示密钥、原始回答、自由文本或模型完整输入输出。

MVP 不引入：

- 数据库。
- 账号系统。
- 云端历史记录。
- Prisma / SQLite。
- 定时任务。
- 通用的信息抓取或监控系统；唯一例外是用户手动触发、仅访问地区官方白名单来源的受控地区检索。

如后续本地 Vercel Functions 开发体验不足，可增加极薄的本地 Node 兼容层，但不得扩大为复杂后端。

## 13.1 MVP 界面基线

- 仅提供浅色模式，不实现深色模式或系统主题跟随。
- 正文和问卷选项的基础字号为 16px，优先保证移动端舒适阅读与触控可用性。
- 报告使用顶部标签在“概览 / 深入分析 / 伴侣讨论”之间切换；不使用底部固定导航或横向滑动作为主导航。

## 14. 专业知识与专项规格前置条件

在开始实现核心问卷、评分和报告前，必须先完成以下文档，并获得用户对实施方案的单独确认。专项规格完成本身不触发项目脚手架、代码编写或依赖安装。

- `specs/research.md`：专业知识来源、关键规则依据、本地政策与成本配置依据。
- `specs/questionnaire.md`：具体问卷模块、题目、选项、跳题逻辑。
- `specs/scoring.md`：红旗规则、维度等级、内部权重、不可展示分数规则。
- `specs/personas.md`：角色化类型、形象边界、类型与维度的映射。
- `specs/report.md`：个人报告和伴侣共同版模板。

这些专项文档必须引用 `specs/research.md`，不能凭感觉设计。

## 15. 非目标

MVP 不做：

- 医疗诊断。
- 法律咨询。
- 心理诊断。
- 明确建议继续或中止妊娠。
- 社区、账号、分享平台。
- 云端敏感数据存储。
- 面向公众大规模发布。
- 自动联网抓取外部信息。

## 16. 验收标准

第一版 MVP 规格完成时，应满足：

- 产品定位和伦理边界清楚。
- 红旗优先级明确。
- 双阶段隐私流程明确。
- 9 个一级评估维度明确。
- 报告结构明确。
- 大模型可用范围和禁止范围明确。
- 默认技术栈和部署策略明确。
- 实现前置专项规格明确。

第一版代码实现前，应满足：

- `specs/research.md` 已完成。
- `specs/questionnaire.md` 已完成。
- `specs/scoring.md` 已完成。
- `specs/personas.md` 已完成。
- `specs/report.md` 已完成。
- 核心规则已有失败测试作为实现起点。
