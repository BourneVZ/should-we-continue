# 002 Core Regression 执行计划

## Summary

目标是把当前实现拉回 `spec.md` / `specs/` 定义的主流程：**核心问卷完成后进入深入推荐页，完成推荐模块或明确跳过后才生成最终个人报告**。同时补齐深入模块完成状态、12 题角色模块、确定性、路径条件和报告可见内容，避免“已生成报告但没有报告内容”以及“深入问卷填写后无影响”。

实施时先将本计划写入 `specs/002-core-regression/plan.md`，然后严格按 Red-Green-Refactor 推进。

## Key Changes

- 核心问卷最后一页不再直接生成 report，而是进入 `deep-dives` 推荐页。
- `deep-dives` 页展示 2-3 个推荐模块 + 固定追加 `persona-deep` 12 题模块。
- 用户完成推荐模块，或点击“跳过全部”并确认后，才生成最终个人报告。
- R3/R4 仍优先进入安全替代页，不展示常规推荐或角色内容。
- 在 workspace 中记录每个深入模块的完成状态，按模块可见题是否全部非 `unanswered` 判定完成。
- 推荐页卡片显示 `未开始 / 进行中 / 已完成`；已完成卡片右下显示完成标记，并允许重新进入修改。
- 未完成模块答案不参与最终报告；完成后重新生成报告时才纳入路径条件、角色映射或确定性。
- 移除核心问卷后默认显示的“已生成初步报告”成功态。
- 最终报告概览必须展示九维总览、优先行动、深入建议/跳过提示；无角色时显示“角色仍在校准中”，不把空角色当作报告完成。
- `ReportDimensionView` 增加 `certaintyLevel` 与 `recommendedModuleId?`，低确定性显示“仍需确认”。
- 路径条件从静态 3+3 扩展到 `scoring.md §15.6` 的 6+6 `PC-*`，按回答谓词生成 `derivedStatus=confirmed|pending`。
- 路径条件增加独立 `readingStatus=confirmed|pending|deferred`，用户标记阅读状态不回写事实、评分、红旗或确定性。
- 推荐排序使用 `scoring.md §15.5`：分数缺口、确定性、R1/R2 加权后取前 3。
- `persona-deep` 始终作为额外推荐项，不占 2-3 个推荐模块名额。
- 只有 `persona-deep` 满足 12 题角色门槛后才显示主/次角色；否则报告显示校准提示。

## Test Plan

- `RuntimeApp`/流程集成：核心问卷完成后进入深入推荐页，不生成 report。
- 深入推荐页：展示 2-3 个推荐模块 + `persona-deep`；跳过全部需要确认。
- 深入模块完成状态：未完成不参与报告，完成后卡片显示完成并可重新生成报告。
- 报告生成：完成推荐模块或确认跳过后才进入三页报告。
- 概览页：不再用“已生成初步报告”掩盖空内容；九维、行动、校准提示必须可见。
- 路径条件：每条 `PC-*` 的 `confirmed/pending` 谓词边界。
- 确定性：缺失/不确定触发 `certaintyLevel=low` 和推荐模块。
- 角色：`persona-deep` 未完成时不显示主角色，完成 12 题且达阈值后显示角色。
- 保留验证：`npm test`、`npm run build`。
- 本地浏览器手工验证：从清空数据开始，完成核心问卷，确认进入深入推荐页；完成一个推荐模块，确认卡片完成态；生成报告后确认报告内容变化。

## Assumptions

- 本轮不引入数据库、账号、云端历史或新依赖。
- 本轮不实现完整伴侣共同讨论重构，只保证个人报告与伴侣页现有边界不被破坏。
- 已有 `buildPersonalReportWorkflow` 作为规格对齐的领域入口，应优先接入运行时，而不是复制一套流程判断。
- `specs/002-core-regression/plan.md` 是本轮执行计划的唯一持久化文档。
